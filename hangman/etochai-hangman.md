# Ревью Hangman 
Проект: https://github.com/ETOOOOOOCHAAAAAAAAAAI/Hangman

Реализация написана в процедурном стиле, весь игровой цикл сосредоточен в одной функции `main`. Есть баги. Игра запускается.

---

## НЕДОСТАТКИ РЕАЛИЗАЦИИ

- Нет отображения уже введённых неправильных букв. Игрок не помнит, что уже вводил.

- Один и тот же неправильный символ можно вводить сколько угодно раз каждый раз программа принимает его и увеличивает счётчик ошибок:

```
У вас осталось 7 возможности ошибиться.
a
У вас осталось 6 возможности ошибиться.
a
У вас осталось 5 возможности ошибиться.
```

- После завершения игры (победа или поражение) нет возможности сыграть снова программа просто завершается.

- Нет приглашения ввода перед запросом буквы. После вывода количества попыток программа молча ждёт ввода:

```
У вас осталось 5 возможности ошибиться.
         здесь просто пустота, непонятно что делать
```

- Правильно введённую букву, уже открытую в маске, можно ввести снова - программа снова напишет "Есть такая буква!!" без какой-либо реакции на повтор.

- Опечатка в сообщении: "Игра **начилась**" вместо "Игра **началась**".

- В приветственном сообщении не закрыта скобка и нет подсказки для выхода:

```go
fmt.Println("Хотите начать игру или выйти?(для старта введите s для выхода q")
```

- Пропущены пробелы в выходных сообщениях:

```go
fmt.Println("Поздравляю вы выиграли!Слово было:")  // нет пробела после "!"
fmt.Println("Ты проиграл.Слово было:")              // нет пробела после "."
```

## ХОРОШО

- Слова читаются из файла правильный подход.
- Корректная обработка ввода: проверяется, что введён ровно один символ и он является буквой.
- Рунная обработка строк (`[]rune`) правильно для работы с кириллицей.
- Картинки висельника аккуратно разделены по стадиям в массиве.

## ЗАМЕЧАНИЯ

### 1. Магическое число `7`

В конце игрового цикла `MaxWrong` объявлен как переменная, но в условии используется магическое число:

```go
var MaxWrong = 7
// ...
} else if wrong == 7 {  // магическое число вместо MaxWrong
```

**Правильно:**
```go
} else if wrong == MaxWrong {
```

### 2. Нейминг

`MaxWrong` — это константа, а не переменная. В Go константы объявляются через `const`, а не `var`. Также по конвенции Go неэкспортируемые идентификаторы пишутся в `camelCase` с маленькой буквы:

```go
var MaxWrong = 7  // неправильно: var, PascalCase

// Правильно:
const maxWrong = 7
```

Однобуквенные переменные `s`, `t`, `ch`, `rs` не несут смысловой нагрузки:

```go
s := scanner.Text()    // что за s?
rs := []rune(s)        // что за rs?
ch := rs[0]            // что за ch?

// Правильно:
input := scanner.Text()
inputRunes := []rune(input)
letter := inputRunes[0]
```

### 3. `stages` определён внутри `main`

Массив картинок висельницы это статические данные, они не должны пересоздаваться при каждом запуске `main`. Вынести на уровень пакета как константу или переменную пакета:

```go
// Неправильно - внутри main():
var stages = []string{ ... }

// Правильно на уровне пакета:
var stages = []string{ ... }

func main() { ... }
```

### 4. `takeRandomWord` возвращает два значения, одно из которых лишнее

```go
func takeRandomWord(words []string) ([]rune, string) {
    idx := rand.Intn(len(words) - 1)
    secret := words[idx]
    secretRunes := []rune(secret)
    return secretRunes, secret
}
```

`string` легко получается из `[]rune` через `string(runes)`, и наоборот. Возвращать оба дублирование и лишние аллокации. Достаточно вернуть `string`, а в вызывающем коде сделать `[]rune(secret)`.

### 5. Off-by-one в `takeRandomWord`

```go
idx := rand.Intn(len(words) - 1)
```

`rand.Intn(n)` возвращает число в диапазоне `[0, n)`. При `len(words) - 1` последнее слово в списке **никогда не будет выбрано**. Правильно:

```go
idx := rand.Intn(len(words))
```

### 6. `rand.Seed` - устаревший вызов

```go
rand.Seed(time.Now().UnixNano())
```

`rand.Seed` помечен как `Deprecated` начиная с Go 1.20. Начиная с Go 1.20 глобальный генератор автоматически инициализируется случайным seed'ом. Строку можно просто удалить.

### 7. Ошибка загрузки файла не прерывает программу

```go
words, err := loadWords("words.txt")
if err != nil {
    fmt.Println("Ошибка чтения файла.", err)
}
```

После печати ошибки программа продолжает выполнение и упадёт на `rand.Intn(len(words))` с паникой, если `words` пустой. Нужно завершать программу. Можно использовать `log.Fatal` 0 он печатает сообщение и вызывает `os.Exit(1)` за одно:

```go
if err != nil {
    fmt.Println("Ошибка чтения файла.", err)
    os.Exit(1)
}

// Или короче:
if err != nil {
    log.Fatal("Ошибка чтения файла: ", err)
}
```

### 8. Висельник не отображается при последней ошибке

При последней (7-й) ошибке:
```go
if !found {
    fmt.Println(stages[wrong])
    wrong++
}
// ...
} else if wrong == 7 {
    fmt.Println("Ты проиграл...")
    break
}
```

Стадия `stages[6]` (полный висельник) **печатается**, но сразу после этого выводится сообщение о проигрыше без финального состояния поля - игрок не видит маску слова в момент поражения.

### 9. Нет подсказки при вводе буквы

```go
for {
    fmt.Println("У вас осталось", MaxWrong-wrong, "возможности ошибиться.")
    scanner.Scan()  // молча ждёт ввода
```

Добавить `fmt.Print("Введите букву: ")` перед `scanner.Scan()`.

### 10. Грамматика в сообщении о попытках

"У вас осталось **7** возможности ошибиться" - "7" и "возможности" не согласуются. При 1 ошибке будет "осталось **1** возможности". Это не критично, но стоит учесть.

## АРХИТЕКТУРА

Всё написано в одной функции `main` - это процедурный стиль. В Go классов нет, но декомпозиция через типы и методы всё равно нужна.

Явно напрашиваются отдельные сущности:

**`Game`** - состояние игры:
- секретное слово и маска
- счётчик ошибок
- множество введённых букв
- методы: `GuessLetter`, `IsWon`, `IsLost`, `MaskString`, `WrongLetters`

**`Gallows`** - стадии рисунка:
- метод `Stage(n int) string`

Сейчас `main` делает всё сразу: загружает слова, инициализирует игру, ведёт диалог, проверяет буквы, выводит рисунок, определяет конец игры. Каждая из этих обязанностей должна быть отдельной функцией или типом.

## Итог

 Базовый функционал работает. Слова читаются из файла - это плюс. Но есть критический баг с повторным вводом неправильных букв, нет возможности сыграть повторно, off-by-one в выборе слова, и весь код живёт в одной функции `main`. Проект нужно дорабатывать.
